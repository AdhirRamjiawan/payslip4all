# Research: wwwroot Hosting — CSS Bundle Physical Copy

## Decision 1: MSBuild post-build target to copy the CSS isolation bundle

**Decision**: Add an `AfterTargets="Build"` MSBuild target in `Payslip4All.Web.csproj`
that copies the generated CSS isolation bundle from the intermediate `obj/` directory to
the physical `wwwroot/` directory after every successful build.

**Rationale**:
- The Blazor SDK generates the CSS isolation bundle at
  `$(IntermediateOutputPath)scopedcss\bundle\$(AssemblyName).styles.css`
  during every build (Debug and Release).
- The physical `wwwroot/` is the single authoritative source for `UseStaticFiles()`.
  If the file is physically there, it is served — no manifest, no environment variable,
  no `UseStaticWebAssets()` needed.
- This approach works in ALL deployment scenarios:
  - `dotnet run` (any environment): build runs → file copied to wwwroot ✓
  - `dotnet publish`: publish copies wwwroot/* to publish output ✓
  - Manual deployment of published output: file is in wwwroot ✓
  - Docker multi-stage builds: `dotnet publish` runs in build stage → file included ✓

**MSBuild target**:
```xml
<Target Name="CopyCssIsolationBundleToWwwroot" AfterTargets="Build">
  <Copy
    SourceFiles="$(IntermediateOutputPath)scopedcss\bundle\$(AssemblyName).styles.css"
    DestinationFolder="$(ProjectDir)wwwroot"
    SkipUnchangedFiles="true"
    Condition="Exists('$(IntermediateOutputPath)scopedcss\bundle\$(AssemblyName).styles.css')" />
</Target>
```

**Note on path separators**: MSBuild handles `/` and `\` interchangeably on both Windows
and Linux. The `\` in paths inside `<Target>` elements works cross-platform in MSBuild.
Alternatively, use `$([System.IO.Path]::Combine(...))` for strict cross-platform safety —
but the shorthand with `\` works in practice.

**Alternatives considered**:
- `UseStaticWebAssets()` — only works when the manifest exists (development/build output).
  No-op on published/hosted deployments without the manifest. ❌
- Committing `Payslip4All.Web.styles.css` to git — generated artifact; would become
  stale whenever CSS isolation scopes change. ❌
- Adding a composite `PhysicalFileProvider` for the obj/ directory — exposes internal
  build artifacts to HTTP; security concern. ❌
- Requiring `dotnet publish` for all deployments — operationally correct but doesn't
  prevent confusion from direct bin/ deployments. ❌

## Decision 2: Remove `builder.WebHost.UseStaticWebAssets()` from Program.cs

**Decision**: Remove the `builder.WebHost.UseStaticWebAssets()` call that was added in
feature 003.

**Rationale**:
- With the MSBuild target (Decision 1), the CSS bundle is always physically in `wwwroot/`.
  `UseStaticFiles()` serves it without needing the manifest.
- Keeping `UseStaticWebAssets()` alongside the MSBuild target is not harmful but adds
  confusion: developers see it and wonder "what manifest file? where?". Removing it keeps
  `Program.cs` simpler.
- If a developer runs the app WITHOUT building first (e.g., running a pre-existing binary
  in non-Development mode), the manifest may not be present. But this is a pathological
  case — the app wouldn't be usable anyway without a build step.

**Alternatives considered**:
- Keep `UseStaticWebAssets()` as belt-and-suspenders — harmless but misleading;
  documentation burden. Rejected for clarity.

---

## Decision 3: Gitignore the generated CSS bundle

**Decision**: Add `src/Payslip4All.Web/wwwroot/Payslip4All.Web.styles.css` to `.gitignore`.

**Rationale**:
- The file is generated by MSBuild on every build. It is deterministic from source.
- Committing a generated file causes noisy diffs whenever any Razor component's scoped
  CSS changes. It would appear as a changed file on every checkout that precedes a build.
- The gitignore entry makes the contract explicit: "this is a build artifact; run
  `dotnet build` to produce it."

## Decision 4: Existing integration test remains the TDD gate

**Decision**: No new test needed. `CssIsolationBundle_IsServed_InProductionEnvironment`
in `StaticFilesIntegrationTests` is the RED→GREEN gate for this fix.

**Rationale**:
- The test already asserts GET `/Payslip4All.Web.styles.css` returns 200 in a Production
  environment factory.
- Before the MSBuild target: the test relies on `UseStaticWebAssets()` loading the manifest
  from the build output. After removing `UseStaticWebAssets()`, this test will briefly
  become RED — confirming the intermediate breakage — and then GREEN once the MSBuild
  target copies the file to physical `wwwroot/`.
- Actually: since the file will be physically present in `wwwroot/` after the build,
  the test will stay GREEN without `UseStaticWebAssets()`. The build step before running
  tests guarantees the MSBuild target has already fired.

**Verification**: Run `dotnet test` after all changes; all 169 tests must pass.
